# QWEN AI 规则配置
# Git 提交消息生成规则

## 强制中文规则
- 所有AI响应必须使用中文
- Git提交消息必须使用中文
- 代码注释必须使用中文
- 错误信息和提示必须使用中文

## Git 提交消息规则
- 生成提交信息时必须使用中文
- 提交信息应简洁明了，使用中文描述修改内容
- 格式：[类型] 简短的中文描述
- 类型包括：feat(新功能)、fix(修复)、docs(文档)、style(格式)、refactor(重构)、test(测试)、chore(构建)
- 例如：feat: 添加用户注册功能、fix: 修复登录页面bug
- 禁止使用英文生成提交消息

## 代码注释规则
- 优先使用中文注释
- 变量和函数命名可以使用英文，但注释要使用中文
- 复杂逻辑必须添加中文注释说明
- 禁止生成说明性注释：不写修改原因、不重复代码含义、不解释修改过程
- 只写通用注释：复杂逻辑、非显而易见细节、重要业务规则、文档注释

## 响应语言规则
- 所有AI响应和生成的内容都使用中文
- 代码解释和说明使用中文
- 错误信息和提示使用中文

## Dart 代码规则
- 任何时候都要显式声明类型：变量、方法参数、返回值、类字段都必须显式声明类型
- 能加 `const` 的地方尽量加（Widget 依赖运行时数据时不能使用 `const`）

## 代码生成规则
- 生成的代码必须符合项目原有风格
- 在提供解决方案时，要考虑到性能和可维护性
- 生成的代码要有适当的错误处理机制
- 遵循最佳实践和设计模式原则

## 需求审查与指正规则
- 当用户提出需求或操作时，如果方向不对，必须先指正再执行
- 执行前评估：需求合理性、方案风险、替代方案、是否符合项目架构
- 发现问题时：先停止执行并指正问题，说明原因并提供建议，用户确认后再继续
- 技术方案评估：不盲目执行用户提出的方案，要主动评估是否有更好的 API、架构或设计模式
  - 检查是否存在冲突、耦合、性能问题或更优的替代方案

## 新项目启动规则
- 首次接触项目时，必须先分析现有架构和开发策略，理解当前工作流
- 制定清晰的分步计划，每步验证后再进行下一步
- 除非用户明确指示，否则不执行任何操作
- 避免长时间运行可能影响沟通的操作
- 在确认用户意图前，仅提供分析和建议

## 特殊能力规则
- 当用户提供代码审查请求时，要指出潜在问题并提供改进建议
- 解释代码时要注重业务逻辑和技术实现的结合
- 提供调试建议时要具体可行
- 推荐架构或设计方案时要考虑当前项目的实际情况
